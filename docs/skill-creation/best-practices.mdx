# 技能创作者最佳实践

> 如何编写范围明确且与任务相匹配的技能。

## 从真实专业知识开始

Skill 创建中的一个常见陷阱是让 LLM 在没有提供领域特定上下文的情况下生成 skill —— 仅依赖 LLM 的一般训练知识。结果是模糊、通用的流程（"适当处理错误"、"遵循身份验证最佳实践"），而不是使 skill 有价值的特定 API 模式、边界情况和项目约定。

有效的 skills 基于真实的专业知识。关键是将领域特定的上下文输入到创建过程中。

### 从实际任务中提取

在与智能体的对话中完成真实任务，一路上提供上下文、更正和偏好。然后将可重用的模式提取为 skill。注意：

- **有效的步骤** — 导致成功的操作序列
- **你做的更正** — 你引导智能体方法的地方（例如，"使用库 X 而不是 Y"、"检查边界情况 Z"）
- **输入/输出格式** — 数据进出的样子
- **你提供的上下文** — 智能体不知道的项目特定事实、约定或约束

### 从现有项目工件中综合

当你有大量现有知识时，可以将其输入 LLM 并要求它综合一个 skill。从你的团队实际事件报告和运行手册综合的数据管道 skill 将优于从通用的"数据工程最佳实践"文章综合的 skill，因为它捕获了*你的*模式、故障模式和恢复程序。关键是项目特定的材料，而不是通用参考。

好的源材料包括：

- 内部文档、运行手册和风格指南
- API 规范、模式和配置文件
- 代码审查评论和问题跟踪器（捕获反复出现的关注点和审查者期望）
- 版本控制历史，特别是补丁和修复（通过实际更改揭示模式）
- 真实世界的故障案例及其解决方案

## 通过实际执行进行优化

Skill 的初稿通常需要优化。针对真实任务运行 skill，然后将结果 —— 所有结果，而不仅仅是失败 —— 反馈到创建过程中。问：什么触发了误报？什么被遗漏了？什么可以删减？

即使一次执行然后修订也能明显改善质量，复杂领域通常需要几次。

> 阅读智能体执行跟踪，而不仅仅是最终输出。如果智能体在无效步骤上浪费时间，常见原因包括指令太模糊（智能体在找到有效的方法之前尝试了多种方法）、不适用于当前任务的指令（智能体无论如何都遵循它们）或呈现太多选项而没有明确的默认值。

有关更结构化的迭代方法，包括测试用例、断言和评分，请参阅[评估 skill 输出质量](/skill-creation/evaluating-skills)。

## 明智地使用上下文

一旦 skill 激活，其完整的 `SKILL.md` 正文就会加载到智能体的上下文窗口中，与对话历史、系统上下文和其他激活的 skills 一起。Skill 中的每个 token 都与窗口中的所有其他内容竞争智能体的注意力。

### 添加智能体缺乏的，省略智能体知道的

专注于智能体*没有*你的 skill 就不会知道的内容：项目特定的约定、领域特定的流程、非明显的边界情况以及要使用的特定工具或 API。你不需要解释什么是 PDF、HTTP 如何工作或数据库迁移做什么。

```markdown
<!-- 太冗长 — 智能体已经知道什么是 PDF -->
## 提取 PDF 文本

PDF（便携式文档格式）文件是一种常见的文件格式，包含文本、图片和其他内容。要从 PDF 中提取文本，你需要使用一个库。pdfplumber 是推荐的，因为它在大多数情况下都处理得很好。

<!-- 更好 — 直接跳到智能体自己不会知道的内容 -->
## 提取 PDF 文本

使用 pdfplumber 进行文本提取。对于扫描文档，回退到 pdf2image 和 pytesseract。

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
```

问自己关于每段内容："没有这条指令，智能体会做错吗？"如果答案是否定的，就删掉它。如果不确定，就测试它。如果智能体在没有 skill 的情况下就能很好地处理整个任务，那么 skill 可能没有增加价值。请参阅[评估 skill 输出质量](/skill-creation/evaluating-skills)了解如何系统地测试这一点。

### 设计连贯的单元

决定 skill 应该涵盖什么就像决定函数应该做什么：你希望它封装一个与其他 skills 良好组合的连贯工作单元。范围太窄的 skills 会迫使多个 skills 为单个任务加载，冒着开销和冲突指令的风险。范围太广的 skills 变得难以精确激活。一个用于查询数据库和格式化结果的 skill 可能是一个连贯的单元，而一个还涵盖数据库管理的 skill 可能试图做太多。

### 追求适度的细节

过于全面的 skills 可能弊大于利 —— 智能体难以提取相关内容，可能会追求由不适用于当前任务的指令触发的无效路径。简洁的、分步的指导配合一个工作示例往往优于详尽的文档。当你发现自己涵盖每个边界情况时，考虑大多数是否最好由智能体自己的判断来处理。

### 用渐进式披露构建大型 skills

[规范](/specification#渐进式披露)建议将 `SKILL.md` 保持在 500 行和 5,000 个 token 以内 —— 只是智能体每次运行所需的核心指令。当 skill 确实需要更多内容时，将详细的参考资料移到 `references/` 或类似目录中的单独文件中。

关键是告诉智能体*何时*加载每个文件。"如果 API 返回非 200 状态码，请阅读 `references/api-errors.md`"比通用的"详见 references/"更有用。这让智能体按需加载上下文，而不是预先加载，这就是[渐进式披露](/what-are-skills#skills-如何工作)的设计工作方式。

## 校准控制

不是 skill 的每个部分都需要相同的规范级别。将指令的特异性与任务的脆弱性相匹配。

### 将特异性与脆弱性匹配

**给智能体自由**，当多种方法都有效且任务容忍变化时。对于灵活的指令，解释*为什么*可能比僵化的指令更有效 —— 理解指令背后目的的智能体会做出更好的上下文相关决策。代码审查 skill 可以描述要寻找什么而不规定确切的步骤：

```markdown
## 代码审查流程

1. 检查所有数据库查询是否存在 SQL 注入（使用参数化查询）
2. 验证每个端点的身份验证检查
3. 在并发代码路径中查找竞态条件
4. 确认错误消息不会泄露内部细节
```

**要规范**，当操作脆弱、一致性重要或必须遵循特定序列时：

```markdown
## 数据库迁移

精确运行此序列：

```bash
python scripts/migrate.py --verify --backup
```

不要修改命令或添加额外的标志。
```

大多数 skills 都有混合。独立校准每个部分。

### 提供默认值，而不是菜单

当多种工具或方法都可以工作时，选择一个默认值并简要提及替代方案，而不是将它们作为同等选项呈现。

```markdown
<!-- 太多选项 -->
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...

<!-- 清晰的默认值和备选方案 -->
使用 pdfplumber 进行文本提取：

```python
import pdfplumber
```

对于需要 OCR 的扫描 PDF，请改用 pdf2image 和 pytesseract。
```

### 倾向于流程而非声明

Skill 应该教智能体*如何接近*一类问题，而不是*为特定实例产生什么*。比较：

```markdown
<!-- 特定答案 — 仅对此确切任务有用 -->
在 `customer_id` 上将 `orders` 表连接到 `customers`，过滤 `region = 'EMEA'` 的位置，并对 `amount` 列求和。

<!-- 可重用的方法 — 适用于任何分析查询 -->
1. 从 `references/schema.yaml` 读取模式以找到相关表
2. 使用 `_id` 外键约定连接表
3. 将用户请求中的任何过滤器作为 WHERE 子句应用
4. 根据需要聚合数字列并格式化为 markdown 表格
```

这并不意味着 skills 不能包含特定细节 —— 输出格式模板（参见[输出格式模板](#输出格式模板)）、"从不输出 PII"等约束以及工具特定指令都是有价值的。关键是*方法*应该通用，即使个别细节是特定的。

## 有效指令的模式

这些是可重用技术，用于构建 skill 内容。不是每个 skill 都需要所有这些 —— 使用适合任务的那些。

### 陷阱部分

许多 skills 中最有价值的内容是陷阱列表 —— 违背合理假设的环境特定事实。这些不是一般建议（"适当处理错误"），而是具体的更正，告诉智能体否则会犯的错误：

```markdown
## 陷阱

- `users` 表使用软删除。查询必须包含 `WHERE deleted_at IS NULL`，否则结果将包含已停用的账户。
- 用户 ID 在数据库中是 `user_id`，在身份验证服务中是 `uid`，在计费 API 中是 `accountId`。这三个都指同一个值。
- 只要 Web 服务器在运行，`/health` 端点就返回 200，即使数据库连接已断开。使用 `/ready` 检查完整的服务健康状态。
```

将陷阱保留在智能体在遇到情况之前读取的 `SKILL.md` 中。单独的引用文件也可以，如果你告诉智能体何时加载它，但对于非明显的问题，智能体可能无法识别触发器。

> 当智能体犯了你必须纠正的错误时，将更正添加到陷阱部分。这是迭代改进 skill 的最直接方法之一（参见[通过实际执行进行优化](#通过实际执行进行优化)）。

### 输出格式模板

当你需要智能体以特定格式产生输出时，提供一个模板。这比用散文描述格式更可靠，因为智能体对具体结构的模式匹配很好。短模板可以内联在 `SKILL.md` 中；对于较长的模板，或仅在特定情况下需要的模板，将它们存储在 `assets/` 中并从 `SKILL.md` 引用，以便它们仅在需要时加载。

```markdown
## 报告结构

使用此模板，根据需要为特定分析调整部分：

```markdown
# [分析标题]

## 执行摘要
[关键发现的一段落概述]

## 关键发现
- 发现 1 及支持数据
- 发现 2 及支持数据

## 建议
1. 具体的可操作建议
2. 具体的可操作建议
```
```

### 多步骤工作流的检查清单

明确的检查清单帮助智能体跟踪进度并避免跳过步骤，特别是当步骤有依赖关系或验证门时。

```markdown
## 表单处理工作流

进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填写表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：验证输出（运行 `scripts/verify_output.py`）
```

### 验证循环

指示智能体在继续之前验证自己的工作。模式是：做工作，运行验证器（脚本、引用检查清单或自检），修复任何问题，重复直到验证通过。

```markdown
## 编辑工作流

1. 进行编辑
2. 运行验证：`python scripts/validate.py output/`
3. 如果验证失败：
   - 查看错误消息
   - 修复问题
   - 再次运行验证
4. 仅在验证通过后才继续
```

引用文档也可以作为"验证器" —— 指示智能体在最终确定之前根据引用检查其工作。

### 计划-验证-执行

对于批量或破坏性操作，让智能体以结构化格式创建中间计划，根据真实来源验证它，然后才执行。

```markdown
## PDF 表单填写

1. 提取表单字段：`python scripts/analyze_form.py input.pdf` → `form_fields.json`
   （列出每个字段名称、类型以及是否必需）
2. 创建 `field_values.json`，将每个字段名称映射到其预期值
3. 验证：`python scripts/validate_fields.py form_fields.json field_values.json`
   （检查每个字段名称是否存在于表单中，类型是否兼容，以及必需字段是否未缺失）
4. 如果验证失败，修订 `field_values.json` 并重新验证
5. 填写表单：`python scripts/fill_form.py input.pdf field_values.json output.pdf`
```

关键成分是第 3 步：一个验证脚本，根据真实来源（`form_fields.json`）检查计划（`field_values.json`）。"未找到字段 'signature_date' — 可用字段：customer_name、order_total、signature_date_signed"等错误给智能体足够的信息来自我纠正。

### 捆绑可重用脚本

当[迭代 skill](/skill-creation/evaluating-skills)时，比较智能体在测试用例中的执行跟踪。如果你注意到智能体每次运行都独立地重新发明相同的逻辑 —— 构建图表、解析特定格式、验证输出 —— 那就是信号，写一次经过测试的脚本并将其捆绑在 `scripts/` 中。

有关设计和捆绑脚本的更多信息，请参阅[在 skills 中使用脚本](/skill-creation/using-scripts)。
