# 优化技能描述

> 如何改进技能的描述，使其在相关提示上可靠地触发。

技能只有在被激活时才有帮助。`SKILL.md` 前置数据中的 `description` 字段是智能体用于决定是否为给定任务加载技能的主要机制。描述不够明确意味着技能在应该触发时不会触发；描述过于宽泛意味着它在不应该触发时会触发。

本指南涵盖如何系统地测试和改进技能描述的触发准确性。

## 技能触发如何工作

智能体使用[渐进式披露](../what-are-skills.md#技能如何工作)来管理上下文。启动时，它们只加载每个可用技能的 `name` 和 `description`——刚好足够知道技能什么时候可能相关。当用户的任务与描述匹配时，智能体将完整的 `SKILL.md` 加载到上下文中并遵循其指令。

这意味着描述承担着触发的全部负担。如果描述没有传达技能何时有用，智能体就不会知道要使用它。

一个重要的细微差别：智能体通常只在需要超出其单独处理能力之外的知识或能力的任务上咨询技能。像"读取此 PDF"这样简单的单步请求可能不会触发 PDF 技能，即使描述完美匹配，因为智能体可以用基本工具处理它。涉及专业知识的任务——不熟悉的 API、领域特定的工作流或不常见的格式——才是写得好的描述可以发挥作用的地方。

## 编写有效的描述

在测试之前，了解好的描述是什么样子的会有所帮助。一些原则：

- **使用祈使语气。** 将描述框定为对智能体的指令："当...时使用此技能"而不是"此技能做..."智能体正在决定是否行动，所以告诉它何时行动。
- **关注用户意图，而不是实现。** 描述用户试图实现什么，而不是技能的内部机制。智能体根据用户要求的内容进行匹配。
- **倾向于主动。** 明确列出技能适用的上下文，包括用户不直接命名领域的情况："即使他们没有明确提及'CSV'或'分析'。"
- **保持简洁。** 几句话到一段短话通常正好——足够长以涵盖技能的范围，足够短以不膨胀跨多个技能的智能体上下文。[规范](../specification.md#description-字段)强制执行 1024 字符的硬限制。

## 设计触发评估查询

要测试触发，你需要一组评估查询——标记为是否应该触发技能的真实用户提示。

```json eval_queries.json
[
  { "query": "我在 ~/data/q4_results.xlsx 中有一个电子表格，C 列是收入，D 列是支出——你能添加一个利润率列并高亮显示低于 10% 的吗？", "should_trigger": true },
  { "query": "将此 json 文件转换为 yaml 的最快方法是什么", "should_trigger": false }
]
```

目标是大约 20 个查询：8-10 个应该触发，8-10 个不应该触发。

### 应该触发的查询

这些测试描述是否捕获了技能的范围。沿几个轴变化它们：

- **措辞**：有些正式，有些随意，有些有拼写错误或缩写。
- **明确性**：有些直接命名技能的领域（"分析此 CSV"），有些描述需求而不命名它（"我老板想要一个来自此数据文件的图表"）。
- **详细程度**：混合简洁的提示和重上下文的提示——短的"分析我的销售 CSV 并制作图表"与带有文件路径、列名和背景的长消息并列。
- **复杂性**：变化步骤数和决策点的数量。包括单步任务与多步工作流，以测试智能体在技能所解决的任务埋藏在更大链中时是否能辨别技能相关。

最有用的应该触发查询是技能会有帮助但连接从查询本身看不明显的那些。这些是描述措辞起作用的案例——如果查询已经要求技能所做的确切内容，任何合理的描述都会触发。

### 不应该触发的查询

最有价值的负面测试用例是**接近失误**——与技能共享关键词或概念但实际上需要不同东西的查询。这些测试描述是否精确，而不仅仅是宽泛。

对于 CSV 分析技能，弱的负面示例是：

- `"写一个斐波那契函数"` —— 明显不相关，什么也测试不了。
- `"今天天气怎么样？"` —— 没有关键词重叠，太容易。

强的负面示例：

- `"我需要更新我的 Excel 预算电子表格中的公式"` —— 共享"电子表格"和"数据"概念，但需要 Excel 编辑，而不是 CSV 分析。
- `"你能写一个 python 脚本读取 csv 并将每行上传到我们的 postgres 数据库吗"` —— 涉及 CSV，但任务是数据库 ETL，而不是分析。

### 真实感提示

真实用户提示包含通用测试查询缺乏的上下文。包括：

- 文件路径（`~/Downloads/report_final_v2.xlsx`）
- 个人上下文（"我的经理让我..."）
- 具体细节（列名、公司名、数据值）
- 随意语言、缩写和偶尔的拼写错误

## 测试描述是否触发

基本方法：通过安装技能的智能体运行每个查询，并观察智能体是否调用它。确保技能已注册并可被你的智能体发现——这因客户端而异（例如，技能目录、配置文件或 CLI 标志）。

大多数智能体客户端提供某种形式的可观察性——执行日志、工具调用历史或详细输出——让你看到运行期间咨询了哪些技能。查看你客户端的文档了解详情。如果智能体加载了你的技能的 `SKILL.md`，技能就触发了；如果智能体在没有咨询它的情况下继续，技能就没有触发。

查询"通过"如果：
- `should_trigger` 是 `true` 且技能被调用，或
- `should_trigger` 是 `false` 且技能没有被调用。

### 多次运行

模型行为是非确定性的——相同的查询可能在一次运行中触发技能，但在下一次不触发。多次运行每个查询（3 个是合理的起点）并计算**触发率**：技能被调用的运行比例。

如果触发率高于阈值（0.5 是合理的默认值），应该触发的查询通过。如果触发率低于该阈值，不应该触发的查询通过。

20 个查询，每个 3 次运行，共 60 次调用。你会想要脚本化这个。以下是一般结构——用你智能体客户端提供的任何内容替换 `claude` 调用和 `check_triggered` 中的检测逻辑：

```bash
#!/bin/bash
QUERIES_FILE="${1:?Usage: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# 此示例使用 Claude Code 的 JSON 输出来检查技能工具调用。
# 用你智能体客户端的检测逻辑替换此函数。
# 如果技能被调用应返回 0（成功），否则返回 1。
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

> **提示**：如果你的智能体客户端支持，你可以在结果明确后提前停止运行——智能体要么咨询了技能，要么在没有它的情况下开始工作。这可以显著减少运行完整评估集的时间和成本。

## 用训练/验证分割避免过拟合

如果你针对所有查询优化描述，你有过拟合的风险——制作一个对这些特定措辞有效但在新措辞上失败的描述。

解决方案是分割你的查询集：

- **训练集 (~60%)**：你用于识别失败和指导改进的查询。
- **验证集 (~40%)**：你搁置的查询，仅用于检查改进是否泛化。

确保两个集都包含应该触发和不应该触发查询的比例混合——不要把所有的正面例子都放在一个集中。随机洗牌并在迭代中保持分割固定，这样你是在比较苹果与苹果。

如果你使用[上面](#多次运行)的脚本，你可以将查询分成两个文件——`train_queries.json` 和 `validation_queries.json`——并分别对它们运行脚本。

## 优化循环

1. **评估**当前描述在*训练集和验证集*上的表现。训练结果指导你的更改；验证结果告诉你这些更改是否泛化。
2. **识别训练集中的失败**：哪些应该触发的查询没有触发？哪些不应该触发的查询触发了？
   - 仅使用训练集失败来指导你的更改——无论你是自己修改描述还是提示 LLM，都将验证集结果排除在过程之外。
3. **修改描述。**专注于泛化：
   - 如果应该触发的查询失败，描述可能太窄。扩大范围或添加关于技能何时有用的上下文。
   - 如果不应该触发的查询错误触发，描述可能太宽泛。添加关于技能*不*做什么的具体性，或澄清此技能与相邻能力之间的边界。
   - 避免添加来自失败查询的特定关键词——那是过拟合。相反，找到这些查询代表的一般类别或概念并解决它。
   - 如果几次迭代后你陷入困境，尝试对描述采用结构上不同的方法，而不是增量调整。不同的框架或句子结构可能在改进无法突破的地方取得突破。
   - 检查描述是否保持在 1024 字符限制之下——描述在优化过程中往往会增长。
4. **重复**步骤 1-3，直到所有*训练集*查询通过或你停止看到有意义的改进。
5. **通过验证通过率选择最佳迭代**——验证集中通过的查询比例。注意最佳描述可能不是你产生的最后一个；早期迭代的验证通过率可能比后来过拟合到训练集的更高。

五次迭代通常足够。如果性能没有改善，问题可能在于查询（太容易、太难或标记错误）而不是描述。

## 应用结果

一旦你选择了最佳描述：

1. 更新 `SKILL.md` 前置数据中的 `description` 字段。
2. 验证描述在[1024 字符限制](../specification.md#description-字段)之下。
3. 验证描述按预期触发。尝试几个提示作为快速健全检查。为了更严格的测试，编写 5-10 个新查询（混合应该触发和不应该触发）并通过评估脚本运行它们——因为这些查询从未是优化过程的一部分，它们给你一个关于描述是否泛化的诚实检查。

前后对比：

```yaml
# 之前
description: 处理 CSV 文件。

# 之后
description: >
  分析 CSV 和表格数据文件——计算汇总统计、
  添加派生列、生成图表并清理杂乱数据。当用户有 CSV、TSV 或 Excel 文件并想要
  探索、转换或可视化数据时使用此技能，即使他们没有
  明确提及"CSV"或"分析"。
```

改进的描述更具体地说明技能做什么（汇总统计、派生列、图表、清理），更广泛地说明它何时适用（CSV、TSV、Excel；即使没有明确关键词）。
