# 评估 skill 输出质量

如何使用评估驱动迭代来测试你的 skill 是否产生良好的输出。

你写了一个 skill，在一个提示上试了一下，它似乎有效。但它是否可靠 —— 在不同的提示上、在边界情况下、比没有 skill 更好？运行结构化评估（evals）可以回答这些问题，并为你提供系统改进 skill 的反馈循环。

## 设计测试用例

测试用例有三个部分：

- **提示**：一个真实的用户消息 —— 某人会实际输入的内容。
- **预期输出**：成功是什么样子的可读描述。
- **输入文件**（可选）：skill 需要处理的文件。

将测试用例存储在 skill 目录内的 `evals/evals.json` 中：

```json evals/evals.json
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "我在 data/sales_2025.csv 中有月度销售数据的 CSV。你能找到收入最高的前 3 个月并制作一个柱状图吗？",
      "expected_output": "一个柱状图图片，显示收入最高的前 3 个月，带有标记的坐标轴和数值。",
      "files": ["evals/files/sales_2025.csv"]
    },
    {
      "id": 2,
      "prompt": "我的下载中有一个叫 customers.csv 的文件，有些行的邮箱缺失 —— 你能清理一下并告诉我有多少缺失吗？",
      "expected_output": "一个清理后的 CSV，处理了缺失的邮箱，加上缺失数量的计数。",
      "files": ["evals/files/customers.csv"]
    }
  ]
}
```

**编写良好测试提示的技巧：**

- **从 2-3 个测试用例开始。** 在看到第一轮结果之前不要过度投入。你可以稍后扩展集合。
- **变化提示。** 使用不同的措辞、详细程度和正式程度。有些提示应该随意（"嘿，你能清理一下这个 csv 吗"），有些应该精确（"解析 data/input.csv 处的 CSV，删除列 B 为空的行，并将结果写入 data/output.csv"）。
- **涵盖边界情况。** 包括至少一个测试边界条件的提示 —— 格式错误的输入、不寻常的请求，或 skill 的指令可能含糊的情况。
- **使用真实的上下文。** 真实用户提到文件路径、列名和个人上下文。"处理此数据"等提示太模糊，无法测试任何有用的内容。

不要担心定义具体的通过/失败检查 —— 只是提示和预期输出。在看到第一轮产生的内容后，你将添加详细的检查（称为断言）。

## 运行评估

核心模式是将每个测试用例运行两次：一次**使用 skill**，一次**不使用它**（或使用以前的版本）。这给你一个比较的基线。

### 工作区结构

在工作区目录中组织评估结果，与 skill 目录并排。每次完整的评估循环都有自己的 `iteration-N/` 目录。在其中，每个测试用例都有一个评估目录，包含 `with_skill/` 和 `without_skill/` 子目录：

```
csv-analyzer/
├── SKILL.md
└── evals/
    └── evals.json
csv-analyzer-workspace/
└── iteration-1/
    ├── eval-top-months-chart/
    │   ├── with_skill/
    │   │   ├── outputs/       # 运行产生的文件
    │   │   ├── timing.json    # token 和持续时间
    │   │   └── grading.json   # 断言结果
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    ├── eval-clean-missing-emails/
    │   ├── with_skill/
    │   │   ├── outputs/
    │   │   ├── timing.json
    │   │   └── grading.json
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    └── benchmark.json         # 汇总统计
```

你手动编写的主要文件是 `evals/evals.json`。其他 JSON 文件（`grading.json`、`timing.json`、`benchmark.json`）是在评估过程中产生的 —— 由智能体、脚本或你产生。

### 启动运行

每次评估运行都应该以干净的上下文开始 —— 没有来自以前运行或 skill 开发过程的残留状态。这确保智能体只遵循 `SKILL.md` 告诉它的内容。在支持子智能体（例如 Claude Code）的环境中，这种隔离是自然的：每个子任务都从头开始。没有子智能体时，为每次运行使用单独的会话。

对于每次运行，提供：

- Skill 路径（或基线无 skill）
- 测试提示
- 任何输入文件
- 输出目录

以下是你给智能体的单次使用 skill 运行的指令示例：

```
执行此任务：
- Skill 路径：/path/to/csv-analyzer
- 任务：我在 data/sales_2025.csv 中有月度销售数据的 CSV。
  你能找到收入最高的前 3 个月并制作一个柱状图吗？
- 输入文件：evals/files/sales_2025.csv
- 保存输出到：csv-analyzer-workspace/iteration-1/eval-top-months-chart/with_skill/outputs/
```

对于基线，使用相同的提示但不使用 skill 路径，保存到 `without_skill/outputs/`。

在改进现有 skill 时，使用以前的版本作为基线。编辑前对其进行快照（`cp -r <skill-path> <workspace>/skill-snapshot/`），将基线运行指向快照，并保存到 `old_skill/outputs/` 而不是 `without_skill/`。

### 捕获时间数据

时间数据让你比较 skill 相对于基线的成本 —— 一个显著提高输出质量但 token 使用量增加三倍的 skill 与一个既更好又更便宜的 skill 是不同的权衡。每次运行完成时，记录 token 计数和持续时间：

```json timing.json
{
  "total_tokens": 84852,
  "duration_ms": 23332
}
```

> 在 Claude Code 中，当子智能体任务完成时，任务完成通知包含 `total_tokens` 和 `duration_ms`。立即保存这些值 —— 它们不会在其他地方持久化。

## 编写断言

断言是关于输出应该包含或实现什么的可验证陈述。在看到第一轮输出后添加它们 —— 你经常直到 skill 运行后才知道"好"是什么样子的。

好的断言：

- `"输出文件是有效的 JSON"` — 可编程验证。
- `"柱状图有标记的坐标轴"` — 具体且可观察。
- `"报告包含至少 3 条建议"` — 可计数。

弱的断言：

- `"输出很好"` — 太模糊而无法评分。
- `"输出使用确切的短语 '总收入：$X'"` — 太脆弱；措辞不同的正确输出会失败。

不是所有东西都需要断言。有些质量 —— 写作风格、视觉设计、输出是否"感觉对" —— 很难分解为通过/失败检查。这些最好在人工审查期间捕获。将断言保留给可以客观检查的内容。

将断言添加到 `evals/evals.json` 中的每个测试用例：

```json evals/evals.json
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "我在 data/sales_2025.csv 中有月度销售数据的 CSV。你能找到收入最高的前 3 个月并制作一个柱状图吗？",
      "expected_output": "一个柱状图图片，显示收入最高的前 3 个月，带有标记的坐标轴和数值。",
      "files": ["evals/files/sales_2025.csv"],
      "assertions": [
        "输出包含一个柱状图图片文件",
        "图表显示恰好 3 个月",
        "两个坐标轴都有标记",
        "图表标题或说明提到收入"
      ]
    }
  ]
}
```

## 评分输出

评分意味着根据实际输出评估每个断言，并记录**通过**或**失败**以及具体证据。证据应该引用或参考输出，而不仅仅是陈述观点。

最简单的方法是将输出和断言给 LLM 并要求它评估每个。对于可以通过代码检查的断言（有效的 JSON、正确的行数、文件存在且具有预期维度），使用验证脚本 —— 脚本比 LLM 判断更可靠，适用于机械检查，并且在迭代中可重用。

```json grading.json
{
  "assertion_results": [
    {
      "text": "输出包含一个柱状图图片文件",
      "passed": true,
      "evidence": "在输出目录中找到 chart.png（45KB）"
    },
    {
      "text": "图表显示恰好 3 个月",
      "passed": true,
      "evidence": "图表显示三月、七月和十一月的柱状"
    },
    {
      "text": "两个坐标轴都有标记",
      "passed": false,
      "evidence": "Y 轴标记为'收入（$）'但 X 轴没有标记"
    },
    {
      "text": "图表标题或说明提到收入",
      "passed": true,
      "evidence": "图表标题为'收入最高的前 3 个月'"
    }
  ],
  "summary": {
    "passed": 3,
    "failed": 1,
    "total": 4,
    "pass_rate": 0.75
  }
}
```

### 评分原则

- **要求通过的具体证据。** 不要给予怀疑的好处。如果断言说"包含摘要"而输出有一个标题为"摘要"的部分，只有一句话含糊的话，那就是失败 —— 标签在那里但实质内容没有。
- **审查断言本身，而不仅仅是结果。** 评分时，注意断言何时太容易（无论 skill 质量如何总是通过）、太难（即使输出很好也总是失败）或无法验证（无法仅从输出检查）。在下次迭代中修复这些。

> 对于比较两个 skill 版本，尝试**盲测比较**：向 LLM 评判展示两个输出，不透露哪个来自哪个版本。评判根据自己的标准对整体质量 —— 组织、格式、可用性、润色 —— 进行评分，不受哪个版本"应该"更好的偏见影响。这补充了断言评分：两个输出可能都通过所有断言，但在整体质量上差异显著。

## 汇总结果

一旦迭代中的每次运行都被评分，计算每种配置的汇总统计并保存到评估目录旁边的 `benchmark.json`（例如，`csv-analyzer-workspace/iteration-1/benchmark.json`）：

```json benchmark.json
{
  "run_summary": {
    "with_skill": {
      "pass_rate": { "mean": 0.83, "stddev": 0.06 },
      "time_seconds": { "mean": 45.0, "stddev": 12.0 },
      "tokens": { "mean": 3800, "stddev": 400 }
    },
    "without_skill": {
      "pass_rate": { "mean": 0.33, "stddev": 0.10 },
      "time_seconds": { "mean": 32.0, "stddev": 8.0 },
      "tokens": { "mean": 2100, "stddev": 300 }
    },
    "delta": {
      "pass_rate": 0.50,
      "time_seconds": 13.0,
      "tokens": 1700
    }
  }
}
```

`delta` 告诉你 skill 的成本（更多时间、更多 token）和它带来的收益（更高的通过率）。一个增加 13 秒但将通过率提高 50 个百分点的 skill 可能是值得的。一个 token 使用翻倍但只提高 2 个百分点的 skill 可能不值得。

> 标准差（`stddev`）只有在每次评估多次运行时才 meaningful。在只有 2-3 个测试用例和单次运行的早期迭代中，关注原始通过计数和 delta —— 当你扩展测试集并多次运行每次评估时，统计度量变得有用。

## 分析模式

汇总统计可能隐藏重要模式。计算基准后：

- **删除或替换在两种配置中总是通过的断言。** 这些没有告诉你任何有用的信息 —— 模型没有 skill 也能很好地处理它们。它们虚增了使用 skill 的通过率，而不反映实际的 skill 价值。
- **调查在两种配置中总是失败的断言。** 要么断言坏了（要求模型做不到的东西），测试用例太难，要么断言检查错了东西。在下次迭代之前修复这些。
- **研究使用 skill 通过但不使用 skill 失败的断言。** 这是 skill 明显增加价值的地方。理解*为什么* —— 哪些指令或脚本起了作用？
- **当结果在运行中不一致时收紧指令。** 如果相同的评估有时通过有时失败（在基准中反映为高 `stddev`），评估可能不稳定（对模型随机性敏感），或者 skill 的指令可能足够模糊，以至于模型每次都以不同的方式解释它们。添加示例或更具体的指导以减少模糊性。
- **检查时间和 token 异常值。** 如果一个评估花费的时间是其他评估的 3 倍，请阅读其执行记录（模型在运行期间所做操作的完整日志）以找到瓶颈。

## 人工审查结果

断言评分和模式分析捕获了很多，但它们只检查你想到要写断言的内容。人工审查者带来全新的视角 —— 捕获你没有预料到的问题，注意到输出技术上正确但偏离了要点，或发现难以表达为通过/失败检查的问题。对于每个测试用例，审查实际输出以及评分。

为每个测试用例记录具体反馈并保存在工作区中（例如，作为评估目录旁边的 `feedback.json`）：

```json feedback.json
{
  "eval-top-months-chart": "图表缺少坐标轴标签，月份按字母顺序而不是时间顺序排列。",
  "eval-clean-missing-emails": ""
}
```

"图表缺少坐标轴标签"是可操作的；"看起来不好"不是。空反馈意味着输出看起来不错 —— 该测试用例通过了你的审查。在迭代步骤期间，将改进重点在你有具体抱怨的测试用例上。

## 迭代 skill

评分和审查后，你有三个信号来源：

- **失败的断言** 指向具体的差距 —— 缺失的步骤、不明确的指令，或 skill 不处理的情况。
- **人工反馈** 指向更广泛的质量问题 —— 方法错了、输出结构不良，或 skill 产生了技术上正确但无用的结果。
- **执行记录** 揭示*为什么*事情出错。如果智能体忽略了指令，指令可能含糊。如果智能体在无效步骤上花费时间，这些指令可能需要简化或删除。

将这些信号转化为 skill 改进的最有效方法是将三个来源 —— 以及当前的 `SKILL.md` —— 给 LLM 并要求它提出更改。LLM 可以综合跨失败断言、审查者抱怨和记录行为的模式，手动连接这些模式会很繁琐。在提示 LLM 时，包含以下指南：

- **从反馈中泛化。** Skill 将在许多不同的提示中使用，而不仅仅是测试用例。修复应该广泛解决根本问题，而不是为特定示例添加狭窄的补丁。
- **保持 skill 精简。** 更少、更好的指令往往优于详尽的规则。如果记录显示浪费的工作（不必要的验证、不需要的中间输出），删除这些指令。如果尽管添加了更多规则，通过率仍然停滞不前，skill 可能过度约束 —— 尝试删除指令，看看结果是否保持或改善。
- **解释原因。** 基于推理的指令（"做 X 因为 Y 往往导致 Z"）比僵化的指令（"总是做 X，绝不做 Y"）效果更好。模型在理解目的时更可靠地遵循指令。
- **捆绑重复的工作。** 如果每个测试运行都独立编写了类似的辅助脚本（
