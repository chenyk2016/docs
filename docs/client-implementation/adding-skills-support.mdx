# 如何为你的智能体添加技能支持

> 为 AI 智能体或开发工具添加 Agent Skills 支持的指南。

本指南介绍如何为 AI 智能体或开发工具添加 Agent Skills 支持。它涵盖完整的生命周期：发现技能、告知模型、将内容加载到上下文中，以及随时间保持内容有效。

核心集成无论智能体架构如何都是相同的。实现细节因两个因素而异：

- **技能在哪里？** 本地运行的智能体可以扫描用户的文件系统以查找技能目录。云托管或沙盒化的智能体需要替代的发现机制——API、远程注册表或捆绑的资产。
- **模型如何访问技能内容？** 如果模型具有文件读取能力，它可以直接读取 `SKILL.md` 文件。否则，你将提供专用工具或以编程方式将技能内容注入提示。

本指南指出这些差异何时重要。你不需要支持每种场景——遵循适合智能体的路径。

**先决条件**：熟悉定义 `SKILL.md` 文件格式、前置数据字段和目录约定的 [Agent Skills 规范](../specification.md)。

## 核心原则：渐进式披露

每个兼容技能智能体遵循相同的三层加载策略：

| 层级 | 加载内容 | 何时 | Token 成本 |
|------|---------|------|-----------|
| 1. 目录 | 名称 + 描述 | 会话开始 | 每个技能约 50-100 个 token |
| 2. 指令 | 完整的 `SKILL.md` 正文 | 当技能激活时 | < 5000 个 token（推荐） |
| 3. 资源 | 脚本、引用、资产 | 当指令引用它们时 | 变化 |

模型从一开始就看到目录，因此它知道有哪些技能可用。当它决定技能相关时，加载完整指令。如果这些指令引用支持文件，模型根据需要单独加载它们。

这保持基础上下文较小，同时让模型按需访问专业知识。具有 20 个安装技能的智能体不会预先支付 20 个完整指令集的 token 成本——仅在给定对话中实际使用的那些。

## 步骤 1：发现技能

在会话启动时，找到所有可用技能并加载其元数据。

### 扫描位置

扫描哪些目录取决于智能体的环境。大多数本地运行的智能体至少扫描两个范围：

- **项目级**（相对于工作目录）：特定于项目或仓库的技能。
- **用户级**（相对于主目录）：给定用户跨所有项目可用的技能。

其他范围也是可能的——例如，管理员部署的组织范围技能，或与智能体本身捆绑的技能。正确的范围集取决于智能体的部署模型。

在每个范围内，考虑扫描**客户端特定目录**和**`.agents/skills/` 约定**：

| 范围 | 路径 | 目的 |
|------|------|------|
| 项目 | `<project>/.<your-client>/skills/` | 客户端的原生位置 |
| 项目 | `<project>/.agents/skills/` | 跨客户端互操作性 |
| 用户 | `~/.<your-client>/skills/` | 客户端的原生位置 |
| 用户 | `~/.agents/skills/` | 跨客户端互操作性 |

`.agents/skills/` 路径已成为广泛采用的跨客户端技能共享约定。虽然 Agent Skills 规范不强制规定技能目录的位置（它只定义里面的内容），但扫描 `.agents/skills/` 意味着其他兼容客户端安装的技能自动对你的可见，反之亦然。

一些实现还扫描 `.claude/skills/`（项目级和用户级）以 pragmatic 兼容，因为许多现有技能安装在那里。其他附加位置包括直到 git 根目录的祖先目录（对 monorepo 有用）、[XDG](https://specifications.freedesktop.org/basedir-spec/latest/) 配置目录和用户配置的路径。

### 扫描什么

在每个技能目录中，查找**包含名为 `SKILL.md` 的子目录**：

```
~/.agents/skills/
├── pdf-processing/
│   ├── SKILL.md          ← 发现
│   └── scripts/
│       └── extract.py
├── data-analysis/
│   └── SKILL.md          ← 发现
└── README.md             ← 忽略（不是技能目录）
```

实用扫描规则：

- 跳过不包含技能的目录，如 `.git/` 和 `node_modules/`
- 可选择尊重 `.gitignore` 以避免扫描构建产物
- 设置合理的边界（例如，最大深度 4-6 级，最多 2000 个目录）以防止大型目录树中的失控扫描

### 处理名称冲突

当两个技能共享相同的 `name` 时，应用确定性优先级规则。

现有实现中的通用约定：**项目级技能覆盖用户级技能。**

在同一范围内（例如，在 `<project>/.agents/skills/` 和 `<project>/.<your-client>/skills/` 下都找到两个名为 `code-review` 的技能），先找到或后找到都可以接受——选择一个并保持一致。发生冲突时记录警告，让用户知道技能被隐藏。

### 信任考虑

项目级技能来自正在处理的仓库，可能是不可信的（例如，新鲜克隆的开源项目）。考虑将项目级技能加载置于信任检查之后——仅在用户将项目文件夹标记为可信时才加载它们。这防止不可信仓库在智能体上下文中静默注入指令。

### 云托管和沙盒化智能体

如果你的智能体在容器或远程服务器中运行，它无法访问用户的本地文件系统。发现需要根据技能范围以不同方式工作：

- **项目级技能**通常是最简单的情况。如果智能体在克隆的仓库上操作（即使在沙盒内），项目级技能随代码一起传播，可以从仓库的目录树中扫描。
- **用户级和组织级技能**在沙盒中不存在。你需要从外部来源提供它们——例如，克隆配置仓库、通过智能体设置接受技能 URL 或包，或让用户通过 Web UI 上传技能目录。
- **内置技能**可以作为静态资产打包在智能体的部署工件中，使它们在每个会话中都可用，无需外部获取。

一旦技能对智能体可用，生命周期的其余部分——解析、披露、激活——工作方式相同。

## 步骤 2：解析 `SKILL.md` 文件

对于每个发现的 `SKILL.md`，提取元数据和正文内容。

### 前置数据提取

`SKILL.md` 文件有两个部分：`---` 分隔符之间的 YAML 前置数据，以及结束分隔符后的 markdown 正文。要解析：

1. 在文件开头找到开头的 `---` 和之后的结束 `---`。
2. 解析它们之间的 YAML 块。提取 `name` 和 `description`（必需），以及任何可选字段。
3. 结束 `---` 之后的所有内容，修剪后，是技能的正文内容。

有关前置数据字段的完整集合及其约束，请参阅[规范](../specification.md)。

### 处理格式错误的 YAML

为其他客户端编写的技能文件可能包含技术上无效的 YAML，但它们的解析器恰好接受。最常见的问题是包含冒号的未引用值：

```yaml
# 技术上无效的 YAML — 冒号破坏解析
description: 使用此技能当：用户询问 PDF
```

考虑一个回退，在重试之前将此类值包装在引号中或将其转换为 YAML 块标量。这以最小成本提高跨客户端兼容性。

### 宽松验证

警告问题，但在可能时仍然加载技能：

- 名称与父目录名称不匹配 → 警告，仍然加载
- 名称超过 64 个字符 → 警告，仍然加载
- 描述缺失或为空 → 跳过技能（描述对披露至关重要），记录错误
- YAML 完全无法解析 → 跳过技能，记录错误

记录诊断以便可以向用户展示（在调试命令、日志文件或 UI 中），但不要因外观问题阻止技能加载。

规范定义了 `name` 字段的严格约束（匹配父目录、字符集、最大长度）。上面的宽松方法故意放宽这些以提高与为其他客户端编写的技能的兼容性。

### 存储什么

至少，每个技能记录需要三个字段：

| 字段 | 描述 |
|------|------|
| `name` | 来自前置数据 |
| `description` | 来自前置数据 |
| `location` | `SKILL.md` 文件的绝对路径 |

将这些存储在以 `name` 为键的内存映射中，以便在激活期间快速查找。

你也可以在发现时存储**正文**（前置数据后的 markdown 内容），或在激活时从 `location` 读取它。存储它使激活更快；在激活时读取它在聚合中使用更少的内存，并捕获技能文件在激活之间的更改。

技能的**基础目录**（`location` 的父目录）稍后需要用于解析相对路径和枚举捆绑资源——在需要时从 `location` 派生。

## 步骤 3：向模型披露可用技能

告知模型有哪些技能可用，而不加载它们的完整内容。这是[渐进式披露的第一层](#核心原则渐进式披露)。

### 构建技能目录

对于每个发现的技能，在适合你的技术栈的任何结构化格式中包含 `name`、`description` 和可选的 `location`（`SKILL.md` 文件的路径）——XML、JSON 或项目符号列表都可以：

```xml
<available_skills>
  <skill>
    <name>pdf-processing</name>
    <description>提取 PDF 文本、填充表单、合并文件。处理 PDF 时使用。</description>
    <location>/home/user/.agents/skills/pdf-processing/SKILL.md</location>
  </skill>
  <skill>
    <name>data-analysis</name>
    <description>分析数据集、生成图表并创建汇总报告。</description>
    <location>/home/user/project/.agents/skills/data-analysis/SKILL.md</location>
  </skill>
</available_skills>
```

`location` 字段有两个目的：它启用文件读取激活（参见[步骤 4](#步骤-4激活技能)），并为模型提供解析技能正文中相对引用的基础路径（如 `scripts/evaluate.py`）。如果你的专用激活工具在其结果中提供技能目录路径（参见[步骤 4 中的结构化包装](#结构化包装)），你可以从目录中省略 `location`。

每个技能向目录添加约 50-100 个 token。即使有几十个安装技能，目录仍然紧凑。

### 放置目录的位置

两种方法很常见：

**系统提示部分**：将目录作为系统提示中的标记部分添加，前面有关于如何使用技能的简要说明。这是最简单的方法，适用于任何具有文件读取工具的模型。

**工具描述**：将目录嵌入专用技能激活工具的描述中（参见[步骤 4](#步骤-4激活技能)）。这保持系统提示干净，并自然地将发现与激活耦合。

两者都有效。系统提示放置更简单、兼容性更广；工具描述嵌入在你有专用激活工具时更干净。

### 行为指令

在目录旁边包含一个简短的指令块，告诉模型如何以及何时使用技能。措辞取决于你支持哪种激活机制（参见[步骤 4](#步骤-4激活技能)）：

**如果模型通过读取文件激活技能：**

```
以下技能为特定任务提供专业指令。
当任务与技能描述匹配时，使用你的文件读取工具加载
列出的位置的 SKILL.md，然后再继续。
当技能引用相对路径时，根据技能的
目录（SKILL.md 的父目录）解析它们，并在工具调用中使用绝对路径。
```

**如果模型通过专用工具激活技能：**

```
以下技能为特定任务提供专业指令。
当任务与技能描述匹配时，调用 activate_skill 工具
使用技能的名称加载其完整指令。
```

保持这些指令简洁。目标是告诉模型技能存在以及如何加载它们——技能内容一旦加载就提供详细指令。

### 过滤

某些技能应从目录中排除。常见原因：

- 用户在设置中禁用了技能
- 权限系统拒绝访问技能
- 技能选择退出模型驱动激活（例如，通过 `disable-model-invocation` 标志）

**从目录中完全隐藏过滤的技能**，而不是列出它们并在激活时阻止。这防止模型浪费回合尝试加载它无法使用的技能。

### 没有可用技能时

如果没有发现技能，完全省略目录和行为指令。不要显示空的 `<available_skills/>` 块或注册没有有效选项的技能工具——这会混淆模型。

## 步骤 4：激活技能

当模型或用户选择技能时，将完整指令传递到对话上下文中。这是[渐进式披露的第二层](#核心原则渐进式披露)。

### 模型驱动激活

大多数实现依赖模型自己的判断作为激活机制，而不是实现工具端触发匹配或关键字检测。模型读取目录（来自[步骤 3](#步骤-3向模型披露可用技能)），决定技能与当前任务相关，并加载它。

两种实现模式：

**文件读取激活**：模型使用目录中的 `SKILL.md` 路径调用其标准文件读取工具。不需要特殊基础设施——智能体现有的文件读取能力就足够了。模型接收文件内容作为工具结果。当模型具有文件访问权限时，这是最简单的方法。

**专用工具激活**：注册一个工具（例如，`activate_skill`），接受技能名称并返回内容。当模型无法直接读取文件时需要，即使可以也有用。相对于原始文件读取的优势：

- 控制返回的内容——例如，剥离 YAML 前置数据或保留它（参见[模型接收什么](#模型接收什么)）
- 在结构化标签中包装内容以在上下文管理期间识别
- 列出捆绑资源（例如，`references/*`）与指令一起
- 强制执行权限或提示用户同意
- 跟踪激活以进行分析

如果你使用专用激活工具，将 `name` 参数约束为有效技能名称集（例如，作为工具模式中的枚举）。这防止模型幻觉不存在的技能名称。如果没有可用技能，根本不要注册该工具。

### 用户显式激活

用户还应该能够直接激活技能，而不等待模型决定。最常见的模式是工具**拦截的斜杠命令或提及语法**（`/skill-name` 或 `$skill-name`）。具体语法取决于你——关键思想是工具处理查找和注入，因此模型接收技能内容而不需要采取激活操作本身。

自动完成小部件（在用户键入时列出可用技能）也可以使这可发现。

### 模型接收什么

当技能激活时，模型接收技能的指令。内容确切样子的两个选项：

**完整文件**：模型看到整个 `SKILL.md`，包括 YAML 前置数据。这是文件读取激活的自然结果，模型读取原始文件。对于专用工具也是有效选择。前置数据可能包含激活时有用的字段——例如，[`compatibility`](../specification.md#compatibility-字段) 说明可以通知模型如何执行技能指令的环境要求。

**仅正文（剥离前置数据）**：工具解析并删除 YAML 前置数据，仅返回 markdown 指令。在具有专用激活工具的现有实现中，大多数采用这种方法——在发现期间提取 `name` 和 `description` 后剥离前置数据。

两种方法在实践中都有效。

### 结构化包装

如果你使用专用激活工具，考虑在识别标签中包装技能内容。例如：

```xml
<skill_content name="pdf-processing">
# PDF 处理

## 何时使用此技能
当用户需要处理 PDF 文件时使用此技能...

[SKILL.md 正文的其余部分]

技能目录：/home/user/.agents/skills/pdf-processing
此技能中的相对路径相对于技能目录。

<skill_resources>
  <file>scripts/extract.py</file>
  <file>scripts/merge.py</file>
  <file>references/pdf-spec-summary.md</file>
</skill_resources>
</skill_content>
```

这有实际好处：

- 模型可以清楚地区分技能指令与其他对话内容
- 工具可以在上下文压缩期间识别技能内容（[步骤 5](#步骤-5随时间管理技能上下文)）
- 捆绑资源向模型展示而不急于加载

### 列出捆绑资源

当专用激活工具返回技能内容时，它还可以枚举技能目录中的支持文件（脚本、引用、资产）——但它应该**不急于读取它们**。模型在技能指令引用它们时使用其文件读取工具按需加载特定文件。

对于大型技能目录，考虑限制列表并注意它可能不完整。

### 权限允许列表

如果你的智能体有控制文件访问的权限系统，**允许列表技能目录**，让模型可以读取捆绑资源而不触发用户确认提示。没有这一点，对每个捆绑脚本或引用文件的引用都会导致权限对话框，破坏包含 `SKILL.md` 之外资源的技能的流程。

## 步骤 5：随时间管理技能上下文

一旦技能指令进入对话上下文，在会话期间保持它们有效。

### 保护技能内容免受上下文压缩

如果你的智能体在上下文窗口满时截断或总结旧消息，**豁免技能内容不被修剪**。技能指令是持久的行为指导——在对话中丢失它们会在没有任何可见错误的情况下静默降低智能体的性能。模型继续操作，但没有技能提供的专业指令。

常见方法：

- 将技能工具输出标记为受保护，以便修剪算法跳过它们
- 使用[步骤 4 中的结构化标签](#结构化包装)识别技能内容并在压缩期间保留它

### 去重激活

考虑跟踪当前会话中已激活哪些技能。如果模型（或用户）尝试加载已在
